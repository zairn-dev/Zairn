# UI監査プロトコル（存在構成概念の提示形式）

## 目的

論文の主張「既存システムは *presence（存在）構成概念* を **事実（fact）** として提示している」を、実際のUIの証拠で根拠づける。各アプリの画面上に現れる「存在の手がかり（presence cue）」を体系的に列挙し、それがどの構成概念（K1〜K5）を、どの提示形式（事実断定／不確実性を明示／生データ）で表示しているかを記録する。

**想定所要時間: 対象アプリがインストール済みの状態で 約1〜2時間。**

---

## 対象アプリ

以下から**アクセスできるものを対象**とする（すべて揃わなくてよい。Zenly後継は入手可能なもの1つを選ぶ）。

1. **Life360**
2. **Find My（iOS の「探す」）**
3. **Snap Map（Snapchat の地図）**
4. **Google Maps 位置情報共有**
5. **Zenly後継アプリ**（利用可能なもの。例: Zenly系の後継/クローン。入手できなければ空欄可）

各アプリで最低1名の友だち/連絡先と位置共有が有効になっている状態を用意する（自分の別端末やテストアカウントで可）。

---

## キャプチャする画面

各アプリについて、少なくとも次の画面をスクリーンショットする。`audit-sheet.csv` にはこれらに対応する行があらかじめ用意されている。

| 画面 | 見るべきもの |
|------|--------------|
| **メインマップ（main map view）** | 友だちのアバター/ピンが地図上のある場所に置かれている表示 |
| **友だち詳細（friend detail）** | 特定の友だちの現在地・住所・場所名・距離を示すカード/シート |
| **場所/到着通知（place / arrival notification）** | 「〇〇に到着」「〇〇を出発」等の場所ベース通知 |
| **ステータス表示（status indicator）** | 最終更新時刻・鮮度・バッテリー・移動/停止・オフライン等の状態表示 |
| **同席の手がかり（co-presence cue）** | 複数人が同じ場所に「一緒にいる」ようにまとめて示す表示 |

1画面に複数のpresence cueがある場合は、**cueごとに1行**記録する（行を追加してよい）。追加分のスクリーンショットは連番（`_02`, `_03` …）にする。

---

## 記録ルール

画面上に見える **presence cue 1つにつき1行**、`audit-sheet.csv` に以下を記入する。

### 1. construct（構成概念）— K1〜K5 のいずれか
そのcueが表現している存在の種類。

| コード | 構成概念 | 意味（UI上の例） |
|--------|----------|------------------|
| **K1** | place（場所） | 「〇〇にいる」— 特定の場所名・住所・地名として位置を提示（例: 「自宅」「渋谷駅」） |
| **K2** | nearby（近接） | 「近くにいる」— 距離・近接（例: 「1.2km先」「近くにいます」） |
| **K3** | co-presence（同席） | 「一緒にいる」— 複数人が同一の場所にいることの提示 |
| **K4** | reachable（到達可能） | 連絡可能・オンライン・応答可能性の提示（例: 「オンライン」「アクティブ」） |
| **K5** | absence（不在/鮮度） | 「いない／オフライン／最後に見たのは〜」— 不在や情報の古さの提示 |

1つのcueが複数の構成概念を担う場合は、主たるものを記入し、`notes` に補足する（必要なら行を分ける）。

### 2. display_form（提示形式）— 3種のいずれか
そのcueが情報を **どれくらい断定的に** 見せているか。**これが本監査の中心変数**。

| 値 | 意味 |
|----|------|
| **asserted-as-fact** | 不確実性を示さず**事実として断定**（例: 「自宅」「〇〇に到着しました」— 推定・誤差・鮮度の注記なし） |
| **hedged-uncertain** | **不確実性・鮮度を明示**（例: 「約」「おおよそ」「最終更新: 5分前」「位置が見つかりません」） |
| **raw-data** | 解釈を加えない**生データ**の提示（例: 地図ピンのみ、座標、タイムスタンプ、速度の数値） |

判断基準: 「そのラベルは、それが *推定であること／古いかもしれないこと* をユーザーに示しているか？」示していなければ **asserted-as-fact**。

### 3. exact_ui_wording（正確なUI文言）
画面に表示されている**文言をそのまま**（原語のまま。日本語UIなら日本語、英語UIなら英語）転記する。アイコンのみで文字がない場合は `(icon only)` と書き、`notes` にアイコンの説明を添える。

### 4. screenshot_filename（スクリーンショットのファイル名）
命名規約: **`app_screen_nn.png`**（例: `life360_mainmap_01.png`）。ファイルは **`docs/ui-audit/screenshots/`** に保存する。`audit-sheet.csv` の各行にはこの規約に沿ったファイル名があらかじめ入っている。cueを追加した行は連番 `nn` を繰り上げる。

- `app` スラッグ: `life360`, `findmy`, `snapmap`, `gmaps`, `zenly`
- `screen` スラッグ: `mainmap`, `frienddetail`, `place_arrival`, `status`, `copresence`

---

## 進め方

1. 対象アプリを開き、上記5画面を順に表示してスクリーンショットを撮る。
2. スクリーンショットを `docs/ui-audit/screenshots/` に規約名で保存する。
3. `audit-sheet.csv` の該当行に `construct` / `display_form` / `exact_ui_wording` / `notes` を記入する。
4. 想定行（app × screen）に無いcueを見つけたら**行を追加**する（`screenshot_filename` は連番で新規に）。
5. 該当画面が存在しない/確認できない場合は、その行の `notes` に理由を書き、他の列は空欄のままにする。

## 出力
- 記入済み `audit-sheet.csv`
- `docs/ui-audit/screenshots/` 内の対応スクリーンショット群

これらから、「presence構成概念が `asserted-as-fact` として提示される割合」を集計し、論文の主張の定量的裏付けとする。
