# SBPP Empirical Evidence — Public Aggregates

Sanitized aggregates supporting the claims in the PoPETs 2027 submission
"Search-Bound Proximity Proofs" Sections 7.4 and 7.5.

Generated: 2026-05-10. Hardware: consumer x86-64 laptop, Node.js v24.

## Files

- `naive-n1000-2026-05-10.json` — Audit re-association attack matrix,
  V1-V7 only (7 protocol variants × 5 attack classes, N = 1000 trials each).
  Generated by an earlier version of `evaluate-sbpp-naive-comparison.mjs`.
- `naive-n1000-with-v8-2026-05-10.json` — Updated attack matrix
  including V8 internalized-token baseline (8 variants × 5 attack classes,
  N = 1000 trials each).
  Generated by an earlier version of
  `packages/geo-drop/test/evaluate-sbpp-naive-comparison.mjs`.
- `naive-n1000-with-v9-2026-05-12.json` — Attack matrix including the
  V9 post-proof signed receipt baseline (9 variants × 5 attack classes,
  N = 1000 trials each), plus an isolated post-proof signing
  micro-benchmark (10,000 iterations, 2,000-iteration warmup).
  Generated by `packages/geo-drop/test/evaluate-sbpp-naive-comparison.mjs`.
- `perf-2026-05-10.json` — Protocol-path microbenchmarks
  (1000 drops × 1000 iterations, 100-iteration warmup).
  Generated by `packages/geo-drop/test/evaluate-sbpp-performance.mjs`.
- `correctness-2026-05-10.json` — Binding correctness across 8 scenarios
  (valid_binding, cross_session, expired, wrong_nonce, wrong_dropId,
  consumption, nonce_replay, parallel_sessions), N = 100 trials each.
  Generated by `packages/geo-drop/test/evaluate-sbpp-correctness.mjs`.
  All 8 scenarios pass expected outcomes (100% expected pass / 100% expected fail).
- `fault-localization-2026-05-10.json` — Fault diagnosis across 3
  fault classes (session_rebinding, result_set_tampering,
  fabricated_context), N = 100 trials each. SBPP distinguishes each
  failure to a specific verification step; V8 returns undifferentiated
  hash mismatch.
  Generated by `packages/geo-drop/test/evaluate-sbpp-fault-localization.mjs`.

## Key Findings

### §7.4 Audit Re-association (RQ2)
Attack class A2 (search-verify decorrelation = audit re-association):

| Variant                          | Adversary wins | Auditor correct attribution |
|----------------------------------|----------------|------------------------------|
| V1 (Plaintext)                   | 1000/1000      | 0/1000                       |
| V2 (GridSE)                      | 1000/1000      | 0/1000                       |
| V3 (App-layer nonce)             | 1000/1000      | 0/1000                       |
| **V4 (SBPP)**                    | **0/1000**     | **1000/1000**                |
| V5 (Signed capability)           | 0/1000         | 1000/1000 (server-side only) |
| V6 (Per-drop permit)             | 0/1000         | 1000/1000 (no ZKP)           |
| V7 (MAC-bound)                   | 0/1000         | 1000/1000 (server-side only) |
| V8 (Internalized-token)          | 0/1000         | 1000/1000 (publicly auditable) |
| V9 (Post-proof signed receipt)   | 0/1000         | 1000/1000 (online server + log finalisation trust) |

V1, V2, V3 are weakly proof-external (Definition 12 / Theorem 13);
re-association succeeds in all 1000/1000 trials, as predicted by the
Separation Theorem. V4 (SBPP), V8 (internalized-token), and V9
(post-proof signed receipt) all block A1–A4 and admit transcript-level
audit. V8 requires circuit modification + trusted-setup rotation; V9
requires the server to be online and willing to sign at proof
acceptance and couples audit integrity to server-side log
finalisation; V4 (SBPP) reuses the existing circuit and finalises the
audit record client-side.

### Isolated post-proof signing micro-benchmark
From `naive-n1000-with-v9-2026-05-12.json` (10,000 iterations,
2,000-iteration warmup, same hardware as performance benchmark):

| Step                                  | Median (μs) | P95 (μs) | P99 (μs) |
|---------------------------------------|-------------|----------|----------|
| V9 isolated post-proof signing only   | 3.0         | 5.5      | 21.0     |

This isolates the SHA-256-over-receipt + log-push cost the V9 server
performs at proof acceptance, above and beyond proof verification.
The substantive operational cost of V9 versus V4 (SBPP) is the
online-server requirement and the 50–200 ms mobile receipt-return
round trip when client-side audit finalisation is required (the latter
not measured here), not the microsecond signing step.

### §7.5 Performance (RQ3)
Protocol-path latency, median over 1000 iterations:

| Scheme              | Median (μs) | Mean (μs) | P95 (μs) | P99 (μs) |
|---------------------|-------------|-----------|----------|----------|
| Plaintext geohash   | 3.8         | 20.9      | 83.8     | 187.3    |
| GridSE              | 696.6       | 767.0     | 1316.5   | 1642.1   |
| **SBPP (Full)**     | **676.4**   | 743.3     | 1228.0   | 1742.7   |

SBPP and GridSE are within measurement noise (~±3%); deterministic
overhead components total ~13 μs (session create 12 μs + digest 0.6 μs
+ validation 0.3 μs).

## Reproduction

```bash
cd packages/geo-drop
node test/evaluate-sbpp-naive-comparison.mjs > naive-results.json
node test/evaluate-sbpp-performance.mjs > perf-results.json
```

Both scripts run on Node.js 24+ with no external services. Results
should be deterministic for the attack matrix (V1-V7) and within
measurement noise for the performance benchmark.
